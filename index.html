HTML

  • The head element contains information about the webpage.
  • The body element represents the visible content shown to the user.
  • Elements nested inside previous elements have a child-parent relationship. Nesting elemnents are the child elements.
  • Questions:
    • Do child elements have a minimum of three?
    • Does a parent element need a minimum of three elements nesting inside?
  • The difference between elements and tags: the element is the entire element from opening to end (including conent?, etc.), the tag is only what appears in the angle brackets

CSS

  • Three CSS Styles
    • CSS inline: made to specific tags in specific HTML file, each tag has separate style attribute
    • CSS Internal Style Sheets: made to header in HTML file, every indicated tage in body of HTML file affected
    • CSS External Style Sheets: made in separate file to apply to all HTML files in project, placed in header as external link

Git

GitHub Description

  • GitHub is a remote cloud space to save and collaborate on projects
  • GitHub stores repositories, or repos, which act as folders for storing files with code and information on a specific project

GitHub Capabilities

  • Adding a repo in GitHub:
    • Click + icon next to profile picture in GitHub account
    • Select New Repository from drop down
    • Fill out repository name, use only lower case letters and dashes for spaces
    • Make public to collaorate, or private to collaborate with only invited coders
    • Add a README file to give information on the project to other developers
    • Add a license, boot camp suggests using the MIT License by default for our purposes
  • Creating Issues
    • Issues help track progress
    • Issue usually contain User Story, which defines goal of user, and Acceptance Criteria, which defines when the goal is reached
      • User Story: "As [User], I want to [Goal]
      • Acceptance Criteria: GIVEN [goal/project], WHEN [this happens], THEN [this results]
    • We created one issue for every language we are introduced to in the prework lesson, as well as one issue for Readme and one for Deployment
    • Written in Markdown language, which was wrongly indicated in lesson, but you can intuit it from selecting font in GitHub
    • Create new issues by going into the repo and navigating to the Issues tab
  • Working in GitHub
  • GitHub Docs on GitHub flow was provided as a link to the workflow of modifying a repo
    • Crete a local feature branches to add to the main branch of code in the repo/project
    • Make commits of the changes made in the feature branch to describe changes in code
    • Push local branch to the remote repo and make a pull request to merge changes to main branch
    • Pull request (PR) is sent to fellow developer to approve changes and give second eye; then merge is accepted
    • Merge the branch after the PR is approved
  • Creating a feature branch to add to main branch in repo
    • navigate to the appropriate directory containing code files and enter "code ." command
    • In coding program, check to make sure all the coding files show up
    • Chech the branch with "git status" command to make sure you are in the main branch
    • Add a feature branch to create new branch for work without changing the main branch using the "git checkout -b [branch]
    • Verify you are in the newly created feature branch with the "git status" command
    • Add all modifications to the newly created feature branch witht the "git add -A" command
    • Document changes with a commit message with the "git commit -m" command and putting the commit message in quotes
    • Check that the main local branch is in synch with remote branch in GitHub with the "git pull origin main" command
    • Push local feature branch to the main branch with the "git push origin [branch]" command
    • Go to repo in GitHub, find newly added feature branch and make pull request for review and merge
    • Main branch should have the modifications made in the local branch after merging

List and Functions of Git Commands:

  • git status: checks what branch we are currently in
  • git checkout [branch name]: navigates to an existing branch
  • git checkout -b [branch name]: creates a new branch and switches to it
  • git add: adds changes to the current branch to the staging area
  • git add -A: add ALL changes to the current branch fo the staging area
  • git commit -m "[commit message]": commits changes to current branch with message (-m) in quotes
  • git pull origin [branch]: make sure current branch is synched with base branch in GitHub. See further notes for compoents:
  • git push origin [branch]: push (modified) branch into remote GitHub branch
    • git: identify git command
    • pull: recieve branch's modifications into local environment
    • push: push modifications from local branch to remote branch
    • origin: source of the pull will be GitHub repo
    • [branch]: the branch pulled from
